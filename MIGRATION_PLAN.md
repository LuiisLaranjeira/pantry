# Pantry — Production migration plan

## Framing the concerns

| Concern | Verdict | What to do |
|---|---|---|
| Expo isn't production-grade | **Misconception.** EAS Build produces real, signed, store-ready binaries. The current stack (SDK 54 + `expo-dev-client`) is the recommended production path. | **Stay on Expo.** Ejecting to bare React Native would cost weeks and buy nothing the current dependencies require. Revisit only if a future native module forces it. |
| Cleaner architecture | **Valid — the real problem.** ~3,700 lines across 8 screens averaging 400+ lines each (`ProfileScreen` 812, `ShoppingScreen` 619, `StockScreen` 546). Screens own UI, state, network, validation, navigation, and business rules at once. | **Restructure aggressively** — feature folders, a data layer, hooks for state, extracted business logic. Details below. |
| RLS / security model | **Addressable without leaving Supabase.** RLS itself is sound; the risk is *policy correctness*. Current policies are reasonable but untested; some privileged ops bypass RLS via edge functions with gaps. | Add an **RLS policy test suite**, move privileged writes behind authenticated edge functions, never let the client touch service-role keys. |
| Vendor lock-in | **Real long-term, low-urgency.** Supabase = Postgres + GoTrue + Storage + Edge Functions. DB and SQL are 100% portable; only auth and edge functions are Supabase-shaped. | **Insert a repository layer.** Screens call `pantryRepo.listStock()`, not `supabase.from('stock_items')...`. Swap-out cost drops to days, not months. |

**Headline recommendation:** Restructure in place. Keep Expo, keep Supabase. The work below makes the codebase production-grade *and* keeps the door open to swap the backend later.

**Key rules this enforces:**
- Screens never import `@supabase/supabase-js`. Only `shared/api/supabaseClient.ts` does. That single chokepoint is what makes a backend swap tractable later.
- Business logic lives in `domain/` as pure functions and is the *only* part with mandatory unit tests.
- Network state (loading/error/cache/invalidation) is TanStack Query's job, not `useState` in every screen.
- `shared/ui` is the design system. No more inline `StyleSheet` per screen for primitives.

## Phased execution (8 phases, each independently shippable)

### Phase 0 — Foundations (no behavior change)
- `src/` skeleton folders created, `tsconfig.json` path alias `@/*`.
- Install (with `npx expo install` where Expo-managed):
  `@tanstack/react-query`, `zod`, `@sentry/react-native`, `expo-application`, `expo-updates`.
- ESLint + Prettier + `eslint-plugin-react-hooks` strict. Husky pre-commit.
- CI: GitHub Actions running `tsc --noEmit`, lint, jest.
- `src/config/env.ts` with zod validation; fail fast at startup if `EXPO_PUBLIC_SUPABASE_URL` etc. are missing.

### Phase 1 — Data layer extraction
Move every `supabase.from(...)` call out of `screens/` into `features/*/api/*Repo.ts`. Each repo function:
- Takes typed args, returns typed results (use `supabase gen types typescript` for `shared/types/database.ts`).
- Throws a tagged `AppError` on failure — never returns `{ data, error }` to callers.
- No React imports.

By feature:
- `stockRepo`: `list(householdId)`, `upsert(item)`, `adjustQuantity(id, delta)`, `delete(id)`, `logAction(...)`
- `shoppingRepo`: `getActiveList`, `addItem`, `toggleChecked`, `complete`, `history`
- `householdRepo`: `create`, `joinByCode`, `members`, `leave`
- `authRepo`: `signIn`, `signUp`, `signOut`, `currentSession`
- `productRepo`: `byBarcode`, `upsertCatalog`

### Phase 2 — TanStack Query everywhere
Replace ad-hoc `useState/useEffect` data fetching with `useQuery` / `useMutation`. One hook per logical operation. Invalidation keys live in `features/*/api/queryKeys.ts`.

Kills ~30–40% of screen code outright (loading state, refresh state, error state, re-fetch wiring).

### Phase 3 — Decompose mega-screens
For each big screen:
- Extract list rows into `components/`.
- Extract modal/sheet flows into their own components.
- Pull form state into a hook (`useShoppingItemForm`).
- Move calculations into `domain/` pure functions with tests.

Order: `ProfileScreen` → `ShoppingScreen` → `StockScreen` → `ReceiptScanScreen` → `ScanScreen`. Target: no screen file > 200 lines.

### Phase 4 — Design system
- `shared/ui` primitives: `Button`, `IconButton`, `TextField`, `Card`, `Sheet`, `EmptyState`, `Skeleton`, `Toast`.
- Theme tokens: spacing scale, type scale, color palette (light + dark).
- Replace inline `StyleSheet.create` for primitives. Keep screen-specific layout styles local.

### Phase 5 — Error handling, observability, resilience
- `ErrorBoundary` at root + per-stack.
- Sentry wired in with release tagging via `expo-application`.
- Structured logger (`shared/lib/logger.ts`) — `info/warn/error`, redact PII, ship errors to Sentry.
- `expo-updates` configured for OTA JS bumps.
- Network retry policy in repos for transient errors only (never on 4xx).
- Offline behavior: at minimum, last-known stock cached via TanStack Query persistence so the app opens without a spinner.

### Phase 6 — Supabase hardening (the security concern, actioned)
- **RLS test suite** in `supabase/tests/`. Spin up local Supabase, seed two households, assert user A can't read/write user B's data. One test per policy.
- **Audit current policies** for known footguns:
  - `products` table is currently world-readable/writable for any authenticated user. Decide: shared catalog or per-household? Document.
  - `households` insert is `with check (true)` — anyone authenticated can create. Add server-side rate limit.
  - No `update`/`delete` policy on `stock_log` — currently immutable, which is good. Document the intent.
- **Move privileged writes behind edge functions** where RLS isn't expressive enough (e.g., joining a household by invite code should be an RPC).
- **Edge function hardening** (`identify-product`, `parse-receipt`):
  - Require a valid user JWT (verify with `supabase.auth.getUser(req.headers.authorization)`).
  - Rate-limit per user (calls paid Groq APIs — abuse vector).
  - Cap image size; validate base64 before forwarding.
- **Secrets**: confirm `GROQ_API_KEY` is only set on the edge function, never bundled. Rotate before launch.
- **Backups**: enable Supabase PITR on production.

### Phase 7 — Environments + release pipeline
- Two Supabase projects: `pantry-dev`, `pantry-prod`. Migrations via `supabase db push` from CI on merge to `main` (dev) and on tag (prod).
- EAS profiles: `development`, `preview`, `production`. Each gets its own `EXPO_PUBLIC_SUPABASE_URL` / `ANON_KEY` via `eas env`.
- Separate bundle IDs for dev/preview (`com.luiislaranjeira.pantry.dev`) so both can be installed.
- `eas submit` configured for TestFlight + Play Internal Testing.
- Versioning: `appVersionSource: "remote"` already set — good.

### Phase 8 — Store readiness
- Privacy policy + terms (static pages; required by both stores).
- App Store data-collection disclosures (email, household data, scanned images).
- Screenshots at required sizes for iPhone 6.7" / 6.5" / 5.5", iPad 12.9", Android phone/tablet.
- Age rating, content rights, demo account for review.
- **Account-deletion flow in-app** (Apple requirement since 2022) — currently missing.

## Out of scope

- Rewriting tests — add tests *as* code is extracted.
- GraphQL / tRPC — repos + TanStack Query is enough.
- Extra state libs — TanStack Query + Context covers it. Zustand only if a real cross-screen need emerges.
- Storage swap (MMKV) — nice-to-have, not a phase-blocker.

|---|---|
| 0 Foundations | 1 day |
| 1 Repos | 2–3 days |
| 2 TanStack Query | 2 days |
| 3 Screen decomposition | 4–6 days (the bulk) |
| 4 Design system | 2–3 days |
| 5 Error/observability | 1–2 days |
| 6 Supabase hardening | 2–3 days |
| 7 Env + release pipeline | 1–2 days |
| 8 Store readiness | 2–4 days (mostly non-code) |

**~3–4 weeks part-time** to ship a v1.0. Compare to ~3–4 *months* to rebuild on bare RN + custom backend, for no functional win.
